인덱스 대시보드 아키텍처 운영 가이드 컨벤션

운영 상태에서 실제 실행으로 이어지는 가이드

대시보드는 현재 상태를 보여주고, 이 문서는 점검과 복구 순서를 제공합니다. 구조는 아키텍처, 정책 원문은 컨벤션을 기준으로 확인합니다.

상태별 대응 사전 점검 핵심 절차 검증/복구

상태별 대응 경로

먼저 대시보드에서 어떤 경고인지 분류하고, 구조 설명이 필요하면 아키텍처로, 금지/허용 기준이 필요하면 컨벤션으로 이동합니다. 이 페이지는 실제 점검 순서만 담당합니다.

Upstream WARN

app upstream가 WARN일 때

먼저 라우팅과 포트 기준이 맞는지 확인한 뒤, 앱과 프록시 순서로 점검합니다.

  1. 아키텍처에서 7101~7103 / 7201~7203 경계를 재확인
  2. Nginx upstream, 앱 컨테이너, health endpoint 순서로 점검
  3. 조치 후 대시보드 데이터를 다시 갱신
대시보드 아키텍처
Exposure WARN

보안 노출이 의심될 때

공개 포트와 bind 주소부터 확인합니다. 정책 확인 없이 방화벽만 만지지 않습니다.

  1. 컨벤션에서 public 허용 포트와 금지 포트 기준 확인
  2. UFW, Nginx ACL, Jenkins loopback bind 상태를 순서대로 점검
  3. Tailscale 접근 경로와 외부 노출 경로를 분리해서 수정
컨벤션 관련 절차
Deploy Gap

서비스와 DB가 미배포일 때

main/dev 분리 원칙을 유지한 채 런타임과 데이터 계층을 순서대로 올립니다.

  1. Docker 런타임과 네트워크 준비 상태 확인
  2. DB/Redis 포트가 7432/7379, 8432/8379 기준인지 확인
  3. 앱 컨테이너 배포 후 Nginx와 healthz를 다시 확인
현재 상태 구조 기준

사전 점검

실제 설치나 점검 전에 로컬 자산, 접근 정보, 원격 선행 검사를 먼저 맞춥니다. 아래 단계는 docs/runbooks/PRECHECK_AND_ACCESS.md를 기반으로 운영 화면용 요약만 추린 것입니다.

1. 로컬 자산 동기화

Windows 기준으로 키와 백업 자산을 먼저 맞춥니다.

powershell -ExecutionPolicy Bypass -File .\scripts\local\prepare-local.ps1
  • .local-assets/keys/J14A408T.pem 존재
  • .local-assets/backup/ssafy-ec2-backup 존재

2. 환경 변수 점검

.env.example를 기준으로 원격 접근과 프록시 항목을 우선 확인합니다.

  • EC2_HOST, EC2_USER, EC2_SSH_PORT
  • NGINX_SERVER_NAME, APP_UPSTREAM
  • ENABLE_JENKINS_PATH, JENKINS_UPSTREAM, JENKINS_ALLOW_CIDR
  • 필요 시 TAILSCALE_AUTH_KEY, ENABLE_SSM

3. 접근/선행 검사

SSH 접속과 원격 preflight가 통과해야 다음 단계로 넘어갑니다.

# Windows OpenSSH ssh -i .\.local-assets\keys\J14A408T.pem ubuntu@<EC2_HOST> # EC2 접속 후 bash scripts/ec2/preflight-check.sh
상태 먼저 보기 정책 확인

핵심 운영 절차

아래 순서는 인프라 기본선과 운영 접근 경로를 복원하거나 새로 맞출 때 사용하는 기본 절차입니다. main/dev 분리와 내부 포트 비공개 원칙은 깨지지 않아야 합니다.

1

SSH 보안 설정

왜 필요한가

root 직접 로그인과 비밀번호 인증을 막아 불필요한 공격 표면을 줄입니다.

어떻게 적용하나

# SSH 설정 파일 확인 sudo vi /etc/ssh/sshd_config PermitRootLogin no PasswordAuthentication no PubkeyAuthentication yes # 문법 검사 후 재시작 sudo sshd -t sudo systemctl restart sshd
2

UFW 방화벽 설정

왜 필요한가

DB, Redis, Jenkins, 내부 앱 포트가 외부에 직접 노출되지 않도록 기본 경계를 만듭니다.

어떻게 적용하나

# 기본 정책 sudo ufw default deny incoming # 공개 포트만 허용 sudo ufw allow 22/tcp sudo ufw allow 80/tcp sudo ufw allow 443/tcp # 필요 시 Tailscale UDP sudo ufw allow 41641/udp sudo ufw enable
3

Docker 런타임 준비

왜 필요한가

FE/BE/AI/DB/Redis를 환경별로 독립 실행하고 main/dev를 네트워크 단위까지 분리하기 위함입니다.

어떻게 적용하나

# Docker 엔진 및 Compose 설치 curl -fsSL https://get.docker.com | sudo sh sudo usermod -aG docker $USER # 확인 docker --version docker compose version
4

Nginx 리버스 프록시

왜 필요한가

Nginx는 유일한 public entrypoint이며, FE/main/dev 라우팅과 Jenkins 접근 제한의 기준점입니다.

어떻게 적용하나

# 예시 라우팅 location / { proxy_pass http://127.0.0.1:7102; } location /dev/ { proxy_pass http://127.0.0.1:7202; } location /jenkins/ { allow 100.64.0.0/10; allow 127.0.0.1; deny all; }
5

Jenkins 운영 접근

왜 필요한가

도메인별 CI/CD를 수행하되 Jenkins 자체는 public service가 아니라 내부 관리 도구로 유지해야 합니다.

어떻게 적용하나

# loopback bind 유지 JENKINS_OPTS=--httpListenAddress=127.0.0.1 # 확인 ss -ltnp | grep 8080 systemctl status jenkins --no-pager
6

Tailscale 운영 채널

왜 필요한가

Jenkins와 기타 내부 관리 도구는 Nginx 제한 또는 Tailscale을 통해서만 접근해야 합니다.

어떻게 적용하나

# 설치 및 접속 curl -fsSL https://tailscale.com/install.sh | sh sudo tailscale up tailscale ip -4

검증 및 복구 후 확인

조치가 끝났으면 시스템 상태와 대시보드 지표를 다시 맞춰야 합니다. 런타임만 정상이고 대시보드가 오래된 데이터면 운영 판단이 어긋납니다.

서비스 상태 확인

systemctl status docker --no-pager systemctl status nginx --no-pager systemctl status jenkins --no-pager systemctl status tailscaled --no-pager

대시보드 데이터 갱신

pwsh -File scripts/local/refresh-infra-dashboard-data.ps1 -HostAlias ssafy-ec2

갱신 후 app upstream, security exposure, jenkins loopback 항목이 기대 상태인지 다시 봅니다.

금지 포트 재확인

5432 / 6379 / 3306 / 27017 / 8080의 public bind는 금지입니다. Jenkins는 항상 loopback이어야 합니다.

앱 포트 7101~7203, 데이터 포트 7432 / 7379 / 8432 / 8379도 외부 공개 대상이 아닙니다.

데이터 분리 검증

main은 7432 / 7379, dev는 8432 / 8379를 사용합니다. cross-environment 접근은 허용하지 않습니다.

구조 기준 정책 기준

실패 시 우선 확인

다음 항목은 실제 장애 대응에서 먼저 보는 체크포인트입니다. 전체 설치 재시도보다 원인 지점을 먼저 좁히는 데 목적이 있습니다.

Jenkins 설치 실패

Java 버전, Jenkins apt repo 키, loopback bind 설정을 먼저 다시 확인합니다.

Tailscale 연결 실패

TAILSCALE_AUTH_KEY 재발급 여부와 기존 노드 중복 등록 상태를 먼저 확인합니다.

SSH 재시작 실패

/etc/ssh/sshd_config.d/99-ssafy-hardening.conf를 점검하고 sshd -t를 통과시키기 전에는 재기동하지 않습니다.

SSM 미연결

EC2 instance profile의 AmazonSSMManagedInstanceCore, 시간 동기화, 에이전트 상태를 함께 봅니다.
