1. Naming Convention

모든 리소스 이름은 이 규칙을 따릅니다. 일관된 네이밍은 자동화와 디버깅의 기반입니다.

리소스패턴예시
EC2 호스트j14a408j14a408.p.ssafy.io
SSH aliasssafy-ec2ssh ssafy-ec2
Docker 컨테이너{service}-{env}be-main, fe-dev, ai-main
Docker 이미지ssafy-{service}ssafy-be:main-latest
DB 인스턴스postgres-{env}postgres-main, postgres-dev
Redis 인스턴스redis-{env}redis-main, redis-dev
Git 브랜치{domain}/{channel}be/main, fe/dev, ai/feature/login
Jenkins Jobssafy-{domain}ssafy-be, ssafy-fe, ssafy-ai
Mattermost 채널{domain}-deploybe-deploy, fe-deploy

2. Port Convention

포트 체계: Main은 71xx / 74xx, Dev는 72xx / 84xx. 외부 노출은 Nginx만.

Public (외부 허용)

PortService비고
80Nginx HTTP항상 열림
443Nginx HTTPS항상 열림
22SSHKey-only 접근
8989Gerrit정책 허용 시만

Internal (127.0.0.1 only)

PortMainDevService
App - BE:7101:7201Backend API
App - FE:7102:7202Frontend
App - AI:7103:7203AI Service
PostgreSQL:7432:8432Database
Redis:7379:8379환경별 공유 Cache/Session
Jenkins:8080CI/CD (공유)

Forbidden (절대 금지)

FORBIDDEN 0.0.0.0 바인딩

5432, 6379, 3306, 27017, 8080 포트의 외부 바인딩 금지.

FORBIDDEN 내부 포트 직접 노출

7101~7203, 7432, 8432 등 내부 포트를 방화벽에서 열지 않음.

FORBIDDEN Cross-env 접근

Main 앱이 Dev DB에 접근하거나 그 반대 금지. 별도 인스턴스, 별도 계정.

3. Docker Strategy

모든 서비스(FE/BE/AI/DB/Redis)는 Docker 컨테이너로 실행합니다. Main과 Dev는 완전 분리.

컨테이너 구성

구분Main PlaneDev Plane
FEfe-main (:7102)fe-dev (:7202)
BEbe-main (:7101)be-dev (:7201)
AIai-main (:7103)ai-dev (:7203)
DBpostgres-main (:7432)postgres-dev (:8432)
Redisredis-main (:7379)redis-dev (:8379)

이미지 태그 정책

태그용도예시
sha-{short}Immutable 식별자 (CI에서 자동 생성)ssafy-be:sha-a1b2c3d
main-latestMain 채널 최신 안정 빌드ssafy-be:main-latest
dev-latestDev 채널 최신 빌드ssafy-be:dev-latest

Docker Compose 원칙

환경 분리

Main과 Dev는 별도 compose 파일 또는 profile로 격리. 네트워크도 분리.

Restart 정책

Main: unless-stopped. Dev: on-failure (실험 환경이라 무한 재시작 방지).

Volume 관리

DB/Redis 데이터는 named volume 사용. 환경별 volume 이름 분리 필수.

4. Branch & Versioning

모노레포에서 도메인별 브랜치를 사용합니다. 버전 태그는 SemVer를 따릅니다.

브랜치 전략

패턴역할배포
{domain}/main안정 릴리스 채널PR merge → auto deploy
{domain}/dev통합 검증 채널push → auto deploy
{domain}/feature/*개발 검증test only, no deploy

{domain} = be, fe, ai. 예: be/main, fe/dev, ai/feature/embedding

버전 태그

채널형식예시
ReleasevX.Y.Zv1.2.0
DevvX.Y.Z-dev.Nv1.3.0-dev.4
RCvX.Y.Z-rc.Nv1.3.0-rc.1
서비스별 태그{domain}-vX.Y.Zbe-v1.4.2, fe-v2.0.1

브랜치 규칙

FORBIDDEN

  • main direct push 금지
  • PR 없이 main merge 금지
  • feature/* 브랜치에서 배포 금지

REQUIRED

  • PR merge 전 CI 통과 필수
  • main 배포 시 버전 태그 필수
  • 통합 릴리스 시 릴리스 노트 작성

5. CI/CD Convention

Jenkins Multibranch Pipeline 기반. 도메인별 독립 파이프라인으로 운영합니다.

파이프라인 구조

파이프라인소스Jenkinsfile 위치
ssafy-beGitLab be/*be/Jenkinsfile
ssafy-feGitLab fe/*fe/Jenkinsfile
ssafy-aiGitLab ai/*ai/Jenkinsfile

배포 트리거

Dev Deploy

*/dev 브랜치 push 시 자동 배포. Docker 이미지 빌드 → dev-latest 태그 → 컨테이너 교체.

Main Deploy

*/main PR merge 시 자동 배포. Docker 이미지 빌드 → sha-* + main-latest 태그 → 컨테이너 교체.

Feature Test

*/feature/* push/PR 시 빌드 + 테스트만 실행. 배포 stage는 skip.

알림

배포 성공/실패 시 Mattermost의 해당 도메인 채널(be-deploy, fe-deploy, ai-deploy)로 알림 전송.

6. Environment & Secrets

시크릿은 .env 파일로 관리하고, Git에는 절대 커밋하지 않습니다.

.env 구조

파일역할Git 추적
.env.example변수 키 목록 (값 없음)O
.env실제 시크릿 값X
.env.tailscaleTailscale 전용 설정X

환경 변수 네이밍

패턴예시설명
{SERVICE}_{ENV}_*DB_MAIN_HOSTMain DB 호스트
{SERVICE}_{ENV}_*DB_DEV_PASSWORDDev DB 비밀번호
REDIS_{ENV}_*REDIS_MAIN_URLMain Redis 접속 URL

Main/Dev 접속정보는 반드시 _MAIN, _DEV suffix로 분리합니다.

금지 사항

NEVER Git에 시크릿 커밋

.env, .pem, *.keys.json, credentials.json 등은 .gitignore에 등록.

NEVER 로그에 시크릿 출력

API 키, DB 비밀번호, 토큰 등을 로그로 찍지 않음. 요청 메타데이터만 기록.

7. Network & Security

최소 권한 원칙. 외부 접근은 Nginx만, 운영 접근은 SSH + Tailscale 이중화.

SSH 강화

  • Password 로그인 비활성화
  • Root 직접 로그인 차단
  • Key-only 인증

방화벽 (UFW)

  • 기본 정책: deny incoming
  • 허용: 22, 80, 443
  • 선택: 8989 (Gerrit)

Nginx Reverse Proxy

  • 모든 외부 트래픽은 Nginx 경유
  • Jenkins는 /jenkins/ + CIDR 제한
  • 보안 헤더 필수 적용

Tailscale VPN

  • Host-level 설치 (1단계)
  • 운영자 사설 접근 이중화
  • Auth key 주기적 회전

Jenkins 접근 제어

  • 127.0.0.1:8080 loopback only
  • Nginx 경유 + CIDR 제한
  • Credential store 우선 사용

Data Isolation

  • Main/Dev DB 완전 분리
  • Main/Dev Redis 완전 분리
  • Cross-env 접근 ACL 금지
