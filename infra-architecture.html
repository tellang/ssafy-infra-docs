1. Network Architecture

외부 트래픽은 Nginx(80/443)를 통해 FE 컨테이너로만 진입합니다. 전체 서비스는 FE → BE → AI → External LLM 순으로 선형적인 요청 흐름을 가지며, Main/Dev 환경은 각각 전용 PostgreSQL과 Redis를 공유합니다.

2. CI/CD Pipeline

Jenkins Multibranch Pipeline은 be/fe/ai 영역을 독립적으로 빌드/배포합니다. 구조 관점에서 중요한 것은 브랜치별로 배포 plane이 달라진다는 점입니다.

Dev Deploy

*/dev 브랜치 push 시 자동 배포. 빠른 통합 검증용 plane으로 흘러갑니다.

Main Deploy

*/main PR merge 시 자동 배포. 승인된 변경만 main plane에 반영됩니다.

Feature Branch

*/feature/*는 build/test만 수행하고 plane 배포로 이어지지 않습니다.

3. Data Isolation

Main과 Dev는 완전히 분리된 Docker 환경을 가집니다. BE와 AI는 각 환경의 PostgreSQL 및 Redis에 접근하고, FE는 DB/Redis에 직접 접근하지 않습니다.

4. Component Responsibilities

Nginx

외부 공개 진입점입니다. 앱과 Jenkins를 직접 공개하지 않고 reverse proxy 역할만 수행합니다.

Application Plane

FE는 사용자 요청 진입점, BE는 비즈니스 로직, AI는 외부 LLM 연동을 담당합니다.

Data Plane

각 환경은 전용 PostgreSQL과 Redis를 갖습니다. 환경 간 cross-access는 허용되지 않습니다.

Infra Plane

Jenkins는 배포 오케스트레이션, Tailscale은 운영 접근 경로를 담당합니다.
